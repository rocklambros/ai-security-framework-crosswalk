## Project Overview

Two-phase graduate project for COMP 4433 Data Visualization (DU, Spring 2026) that also produces a reusable community artifact for the AI security standards ecosystem.

- **Project 1 (midterm, 30%):** Exploratory visual analysis of the crosswalk graph using matplotlib/seaborn. Scientific audience. Notebook deliverable.
- **Project 2 (final, 30%):** Interactive Dash app for navigating cross-framework mappings. User-facing, deployment-ready. GitHub repo deliverable.

Both projects share the same underlying dataset: a graph of nodes (controls, risks, techniques, requirements) and edges (mapping relationships) across 10 AI security frameworks.

## Mapping Engine Design (Next Conversation Topic)

### What holds from the AIUC-1 pipeline

- Multi-signal composite architecture (reference bridge + semantic + keyword + function match)
- Semantic similarity with Z-score normalization per row (solves the high-baseline problem)
- TF-IDF with bigrams, sublinear_tf, and synonym expansion
- Rationale taxonomy (8 codes) and relevance classification (Primary/Secondary)
- Confidence tiers (Direct, Related, Tangential)
- Multiplicative function match boost (amplifies, doesn't create signal)

### What breaks at scale

1. **Reference bridge (Signal 1, weight 0.467) is AIUC-1/OWASP-specific.** It computes Jaccard overlap on shared OWASP LLM Top 10 citations. For pairs where neither framework cites LLM Top 10, this signal returns zero. Recommended fix: generalized reference bridge via the graph itself (shared-neighborhood Jaccard on graph adjacency).
2. **Function match assumes one side is risks, the other is controls.** Control-to-control and technique-to-risk pairs need different boost logic.
3. **10 anchor pairs are pair-specific.** New pairs need new anchors before running the pipeline.
4. **Thresholds calibrated on one dataset.** 14.3% generalization gap on NIST validation. Need per-pair calibration with anchor holdout.
5. **Signal weights may need per-pair tuning.** Weight stability analysis (perturb ±15%, measure mapping changes) should gate whether pair-specific weights are needed.

### Anti-overfitting strategies

1. Per-pair anchor holdout (create 10, hold out 3, calibrate on 7, validate on 3)
2. Signal weight stability analysis (±15% perturbation, report % of mappings that change tier)
3. Cross-validation across framework pairs (hold out one pair, train on others)
4. Semantic model selection per domain distance (regulatory vs technical language)
5. Synonym group expansion as living document

### Proposed architecture

```
mapping_engine/
├── config/
│   ├── pairs/{source}_{target}.yaml    # Per-pair: weights, thresholds, anchors, bridge_standard
│   ├── synonyms.yaml                    # Shared synonym groups
│   └── function_profiles.yaml           # Threat/control function profiles (all frameworks)
├── engine/
│   ├── signals.py                       # Framework-agnostic signal computation
│   ├── bridge.py                        # Generalized reference bridge (graph-based)
│   ├── taxonomy.py                      # Function classification (expanded)
│   ├── mapper.py                        # Pair-agnostic orchestrator
│   └── calibration.py                   # Anchor validation + threshold tuning
├── scripts/
│   ├── run_pair.py                      # Map one pair
│   ├── run_all.py                       # Map all configured pairs
│   └── add_pair.py                      # Scaffold new pair config
└── tests/
```

## Key Technical Decisions

- **Graph data model** over pairwise crosswalk files. NetworkX in-memory graph, serialized as nodes.json + edges.json.
- **EU GPAI Code of Practice** over EU AI Act directly. The CoP has discrete mappable commitments and measures. The Act's articles are too broad.
- **OWASP COMPASS dropped.** It's a methodology/spreadsheet tool, not a control framework with discrete mappable entries.
- **CSA AICM xlsx files gitignored.** License doesn't permit redistribution of raw spreadsheet. Derived JSON/markdown is fine.
- **Confidence gradient on edges.** authoritative > expert > inferred > unvalidated. Drives visual weight in the Dash app.
